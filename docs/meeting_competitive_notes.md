# Competitive color notes (10 min) — owner meeting

Credibility notes only. **Do not promise competitor feature parity.**
Prepared for the J2 4–5pm conversation.

---

## Positioning line

Ekahau is the **survey instrument**. J2’s product is the **client-ready deliverable
workflow** — field photos + exact-name merge + Drafter hard-fail resolution +
branded layered Word — without requiring techs to live inside Ekahau reporting.

---

## What Ekahau already does (acknowledge, don’t trash)

- **Template / custom reporting** in AI Pro (and cloud reporting updates): loops,
  data tags, heatmaps, DOCX/PDF, branding.
- **AP note images** in survey apps; template tag `ap-note-image` can pull AP-linked
  photos into reports when photos were attached *inside* the Ekahau project.
- Community scripts unzip `.esx` to extract AP pictures after the fact — proof that
  “photos exist but packaging them for the client is still friction.”

---

## Where J2’s tool still wins the room

| Pain | Our answer |
|---|---|
| Techs hate living in heavy survey software for as-built photos | Phone browser capture — Close+Far only, zero install |
| Photos taken outside Ekahau never join cleanly | Exact AP name join + loud flags |
| Incomplete jobs become “final” Word assemblies | Readiness gate + manual Push merge |
| Hand-assembly is haphazard | Repeatable Job pipeline + override audit |
| Exec vs engineer readers | Layered spine (Summary / Findings / Appendices) |

If Josh says “Ekahau can already put photos in reports”:  
*Yes — when photos are captured and attached inside their ecosystem. Our Tech path
is deliberately lighter, and the Drafter gets hard-fail mismatch handling plus
bulk override reasons before anything ships.*
